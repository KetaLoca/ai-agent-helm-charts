> What docs exist, what each contains, and how they stay correct.

## 2. Root README outline

1. One-line description + **disclaimer block** (verbatim from `00` §5).
2. Charts table: name, what it deploys, status (alpha), Artifact Hub/OCI link.
3. **Quick start (safe by default):** `helm install` from OCI + immediate `kubectl port-forward` (NOT ingress) to reach it.
4. Security posture in 5 bullets (no public exposure, external secrets, single-writer, pinned images, hardened context) → link to `docs/security.md`.
5. Compatibility matrix per chart.
6. Links: per-chart READMEs, docs, contributing, security, license.
7. "Unofficial / not affiliated" reiterated near the top and bottom.

## 3. Per-chart README outline

1. Disclaimer + what it deploys + upstream links.
2. **TL;DR install** (OCI) and **port-forward** access.
3. Prerequisites (Hermes: a StorageClass for RWO; OpenClaw: **operator + CRDs installed**, K8s ≥ 1.28).
4. **Values reference table** — generated by `helm-docs` from `values.yaml` comments; CI fails if stale (`05` §2).
5. Key design decisions & defaults (link to `02`/`03`): single-writer, Recreate, API key requirement, ingress off, CRD passthrough/`extraSpec`.
6. Common recipes (link examples): minimal, production, private access, external secrets.
7. **Security & risks** section (from chart spec §17/§11).
8. **Compatibility table:**
   - Hermes: chart version ↔ `appVersion` (Hermes image) ↔ min K8s.
   - OpenClaw: chart version ↔ target CRD/operator version ↔ OpenClaw app version ↔ min K8s. State the `extraSpec` forward-compat policy.
9. Backup/restore + upgrade pointers.
10. Uninstall behavior (retained PVCs).

## 4. Quick-start vs production-like

- **Quick start** must model the *safe* path: install → `port-forward` → set API key via a dev secret → reach the gateway locally. Never show a public Ingress as the first experience.
- **Production-like** guide (in `docs/` + `production-values.yaml`): external secrets, pinned digest, NetworkPolicy egress allow-list, resource sizing, backups, identity-aware access, monitoring. Cross-links the checklist.

## 5. Docs that must include explicit warnings

- Any mention of Ingress/dashboard → the exposure warning + recommended alternatives.
- `secrets.create`/inline keys → "dev only, ends up in the release".
- `chromium`/`webTerminal`/`selfConfigure`/`autoUpdate` → capability/security implications.
- NetworkPolicy → L4-only limitation (can't allow-list hostnames).
- Uninstall → PVC retained on purpose; how to actually delete data.

## 6. Keeping docs correct (anti-rot)

- **helm-docs** generates the values tables from `values.yaml` `# --` comments; `lint-test.yaml` fails on drift.
- **Examples are CI-rendered** (`06` §2) — if an example breaks, CI fails before docs mislead.
- **Compatibility tables** updated on every `appVersion`/CRD bump (part of the release checklist).
- Disclaimer is centralized (a snippet) and referenced, so it can't drift between docs.
- A docs PR template checkbox: "updated compatibility table / examples / helm-docs if values changed".

## 7. Versions & i18n

- Docs target the latest chart minor; `upgrade.md` carries historical migration notes.
- **Language:** English (matches public repo + Artifact Hub). A Spanish translation of the root README/quick-start is a nice-to-have, **not** v0.x scope (avoid divergence). **Open question for maintainer** (`08`).

## 8. Out of scope (docs)

- A docs website/MkDocs site (markdown in-repo + Artifact Hub is enough for v0.x).
- Tutorials for specific LLM providers beyond a pointer to set the right env/secret.
